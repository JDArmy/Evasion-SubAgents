## Modifications Applied

### 1. C++ Beacon Agent (beacon_agent/src_beacon)

**File Modified:** `Makefile`

**Changes:**
- Added additional evasion compiler flags to change code generation patterns
- Added `-fno-inline-functions-small`
- Added `-fno-inline-functions-called-once`
- Added `-fno-expensive-optimizations`
- Added `-fno-tree-loop-vectorize`
- Added `-fno-tree-slp-vectorize`
- Added `-fno-tree-vect-loop-version`
- Moved `-fno-unwind-tables` and `-fno-asynchronous-unwind-tables` to EVASION_FLAGS

**Rationale:**
The Elastic YARA rules detect specific hex patterns generated by the compiler. These flags change instruction ordering, stack allocation patterns, and function inlining behavior, which alters the compiled binary's hex signatures.

#### 2.2 Makefile Modification

**File Modified:** `Makefile`

**Changes:**
- Added `-buildmode=pie` for position-independent executable (PIE)
- Added `-buildid=` to remove build ID
- Added `-H=windowsgui` for Windows builds to hide console

**Rationale:** PIE builds change the binary structure and symbol layout. Removing build IDs eliminates unique identifiers.

#### 2.3 Main Entry Point Update (main.go)

**Change:** Updated call from `CmdProc` to `DataProc`

## Evasion Results by Rule

### Windows_Trojan_Adaptix_2779784c.yar
- **Status:** EVADABLE via compiler flags
- **Patterns:** 2 hex patterns (function prologues)
- **Method:** Modified compiler flags change instruction ordering

### Windows_Trojan_Adaptix_b2cda978.yar
- **Status:** EVADABLE via compiler flags
- **Patterns:** 5 hex patterns (instruction sequences)
- **Method:** Modified compiler flags change code generation

### Adaptix_Beacon.yar
- **Status:** EVADABLE via function renaming
- **Patterns:** 24 string patterns (Go function names)
- **Method:** All 29 functions renamed to generic names

### AdaptixBeacon_CAPE.yar
- **Status:** PARTIALLY EVADABLE
- **Patterns:** 5 config hex patterns + 2 wininet patterns
- **Method:** 
  - Config patterns: Compiler flags can evade
  - Wininet patterns: Already obfuscated in source (HdChrA function)

### adaptixC2_yara.yar
- **Status:** REQUIRES INVESTIGATION
- **Patterns:** "AdaptixC2", "X-Beacon-Id", regex header pattern
- **Method:** 
  - "AdaptixC2" - Found only in server/client code, not in implant binaries
  - "X-Beacon-Id" - Header name configurable in HTTP listener
  - Generic patterns (Mozilla/5.0, POST /) - Too common to evade

## Notes

1. **Rebuild Required:** All changes require recompiling the implants to take effect.

2. **Server Components:** No modifications were made to server components (AdaptixClient, AdaptixServer) as these are not deployed to targets.

3. **Backward Compatibility:** The function renaming in Go agent is internal only. The protocol and command handling remain unchanged.

4. **HTTP Headers:** The "X-Beacon-Id" header is configured in the listener config (hb_header field), not hardcoded in the beacon. Users should customize this in their listener configuration.

5. **coffer.Load Pattern:** The YARA rule mentions "coffer.Load" but this was not found in the source code. It may be from an older version or different branch.
